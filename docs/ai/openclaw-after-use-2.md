# openclaw 使用体验（二）

一个字——拉。

为什么我和上一篇的评价会出现如此大的反差？原因很简单：当它真正开始“干活”的时候，问题就暴露无遗，错误频出，稳定性堪忧。

前面提到过，它最大的优势在于拥有完整的服务器权限。只要给它一个完整的运行环境、充足的 token 额度和足够的时间，它理论上具备相当强的执行能力。我甚至认为，在理想条件下，它可以完成世界上 60% 以上的自动化任务。

但问题在于，“能做”不等于“做好”。真正进入复杂任务阶段时，它的执行质量、错误控制能力以及自我修正能力，都远远没有达到可以放心托付生产环境的程度。

能力的上限很高，但下限也同样令人不安。下面我来讲讲他的问题（个人理解）：

##### （一）记性堪忧

openclaw 并没有采用向量数据库来存储任务或长期记忆，而是通过生成和维护 Markdown 文档的方式进行持久化。这种设计在结构上看似简单直接，但在实际使用过程中暴露出明显的问题。例如，我原本希望 ClawBot 能够每天早上定时推送面经内容。前一天对任务进行明确安排时，一切看起来都没有问题。然而到了第二天，它却完全“失忆”——既没有触发任务，也无法正确回忆此前的约定。这说明基于 Markdown 文件的持久化方式，在长期记忆管理和任务连续性方面存在明显不足。缺乏向量检索或语义级记忆索引，使得任务的上下文关联能力较弱，最终影响了系统的可靠性。

##### （二）token消耗成本非线性

这一点，相信不少开发者都会深有体会。

我使用的是国产模型 Minimax。调教一个稍微复杂一点的任务，往往需要反复试错和多轮交互，光是前期调试成本就可能花掉几十块人民币。理论上，一旦任务调教稳定，单次运行成本其实并不高，大概几毛钱一次，算是可以接受。

问题在于，它并不稳定。

任务设定好之后，没过多久就开始“失忆”，逻辑断裂、上下文丢失，甚至连之前确认过的流程都会重新走偏。于是不得不重新调教、重新消耗 token。来回折腾两天，前前后后花了一百多块钱，结果依然达不到预期的稳定效果。

这种体验最让人沮丧的地方在于：成本并不是线性增长的，而是被“不稳定性”反复放大。原本可以沉淀为长期可复用能力的调教过程，却因为记忆机制和执行可靠性问题，变成了反复推倒重来的消耗。还不如我那套“古法爬虫服务”。至少后者逻辑透明、行为确定，不会突然“忘事”，更不会在关键时刻自我偏离。

##### 总结

目前来看，openclaw 确实有它的亮点。经过一段时间的研究和实际使用，我认为它最大的优势主要体现在两个方面。

- 它拥有对服务器或本地环境的完整权限。只要给予足够的授权，它几乎可以像一个“自动化运维助手”一样，自由安装依赖、配置环境、编写脚本并执行任务。这种能力上限是非常高的——理论上，只要模型能力足够、上下文足够长，它可以完成大量复杂的自动化流程。

- 它背后有 ClawHub 的生态支持，里面集成了大量现成的 skills。对于常见场景，确实可以快速拼装能力模块，降低初期搭建成本。

但问题也同样明显。

- 成本问题。token 消耗非常快，尤其是在调教复杂任务时，前期的试错成本远高于预期。模型的不稳定性又进一步放大了这种消耗——每一次“记忆断裂”或逻辑偏移，都意味着重新沟通、重新消耗。

- 记忆机制不稳定。它经常“忘记”之前任务是如何设计和运行的。所谓的定时任务，本质上只是定期向模型发送一段预设 prompt，再让模型重新推理一遍。这种机制本身就依赖模型当下的状态和上下文质量，稳定性天然不足。一旦上下文环境变化，或者历史记录未被正确加载，结果就可能偏离预期。换句话说，它更像是“每次重新思考一次任务”，而不是“稳定执行既定逻辑”。

从实验角度来看，它是一个非常有潜力的工具；但从生产可控性和成本效率角度来看，目前还需要更多打磨。尤其是在长期记忆与任务持久化方面，或许需要更成熟的存储方案，比如更合理的状态管理机制，甚至引入向量数据库或结构化任务编排系统。

我的建议是：可以持续关注，但暂时不必重仓投入。等生态和底层记忆机制更成熟后，再决定是否全面采用。
