# RAG全流程

> 新的方案不一定合适，但是成熟的方案一定能用

### rag介绍

知识库 Agent 的核心本质其实就是 **RAG（Retrieval-Augmented Generation，检索增强生成）** 的应用过程。RAG 是构建 **智能客服、企业知识库问答系统以及产品问答助手** 的关键技术之一。

当我们希望 AI 能够回答特定领域的问题（例如公司产品手册、内部技术文档等）时，如果直接将大量长文本输入给模型，不仅会受到 **模型上下文窗口大小的限制**，还会带来 **推理成本高、响应速度慢以及回答准确率下降** 等问题。

RAG 通过 **“先检索、再生成”** 的方式有效解决了这一问题：系统首先从外部知识库中检索出与用户问题最相关的文档片段，然后将这些内容作为上下文提供给大模型生成最终答案。这样既能降低模型的输入规模，又能显著提升回答的准确性与可控性。

---



### rag流程

总体分为**提问前**和**提问后**。

#### 提问前

提问前做的是数据准备的工作，分为：**分片**、**embedding**、**存储**三个部分。

- 分片：将完整的文档切割为多个片段，让后续的检索更加精确。

  - > 需要保证每个片段的语义完整，避免信息割裂。

- embedding：用专门的embedding模型将文本片段转为向量。

  - > 原理是语义相近的文本向量距离更近。

- 存储：将文本片段以及对应的向量存入向量数据库，方便后续快速查询。

  - > 需要存储向量和原始文本。



#### 提问后

提问后的链路主要做的是回答生成，主要分为**召回**、**重排**，**生成**三个部分。

- 召回：当用户提问的时候，从向量数据库召回相关片段。
  - 1. 将用户的问题通过embedding模型转化为销量。
    2. 用向量相似度的算法计算问题的向量和数据库中片段的向量相似度，从中挑选出TopK个最相关的片段。
- 重排：对召回的片段进一步精筛，用专门的计算文本对相似度的模型，逐对计算问题与召回片段的语义相关性，从上一轮的片段里选出n个最相关的片段。
- 生成：让ai生成答案。
