# kafka疑难杂症



## mq消息积压

> 重点是做好监控和上线之前压测

### 如何发现消息发生积压

需要依赖**监控系统**（比如普罗米修斯和grafana）。重点关注：

- 待消费的数量

- 最后消费时间
- 消费速度/生产速度比值
- 消费端异常告警

根据上面的指标设置阈值，达到阈值进行告警。

### 常见原因和解决方案

大概原因是消费端速度**跟不上**生产的速度。

> 还有可能是生产者重复生产消息。

解决方案：

- 优化消费者端代码
- 扩充消费者数量
- 消费者二次转发（自己思考出来的

### 已经出现了堆积怎么解决

思路是先保证业务可以正常运行，之后再慢慢处理堆积的消息。比如可以先将堆积的消息移到另外的消息队列，保证当前的队列中，新来的任务可以被正常处理；之后再优化消费者，消化历史消息。



## mq可靠性保证

分为三个部分去考虑，生产者，broker，消费者

### 生产者

#### 本地消息表方案

推荐使用**最终一致性方案**的思想。

1. 需要在业务数据库加一个**本地消息表**`t_msg_record`：

```json
id
body:消息体
status：消息状态 -- 0:待投递 1:投递成功 2:投递失败
```

用这种方式就利用了数据库的事务，把业务操作和消息记录作为了一个原子操作，业务成功之后，消息记录也一定会有。

2. 此外还需要一个**job**对待发送的消息进行补偿。可以采用衰减的方式进行投递（比如可以指数退避）。

#### kafka自身支持

kafka的生产端需要考虑发送给集群的时候失败。为了保证消息不丢失，一般会建议使用`producer.send(msg,callback)`，同时可以给producer设置一些参数，来提升发送的成功率。

>  Ack=-1表示leader和follower同时接受确认。

### broker

Kafka的集群有一些机制来保证消息的不丢失，比如复制机制，持久化机制和ISR机制。

- 使用持久化存储，一旦写入磁盘，宕机也会有数据。
- 使用复制机制来保证数据的可靠性，每个分区有多个副本，副本可以分布在不同的节点，当一个节点宕机，其他节点的副本仍然可以提供服务，保证消息不丢失。

### 消费者

使用手动提交偏移量的方式，避免拉取消息之后，没处理完就提交。
