# clickhouse和doris对比

> 这篇博客（https://cloud.tencent.com/developer/article/2506311）总结的很全，我就在此基础上，汇聚高见，站在前人肩膀上壮大几身，

## 一、选择角度

### （一）架构复杂度对比

| 对比维度       | Doris                                 | ClickHouse                                  |
| -------------- | ------------------------------------- | ------------------------------------------- |
| 架构模式       | **FE + BE 分离架构**                  | **单机架构为核心**                          |
| 核心组件       | FE（Frontend）+ BE（Backend）         | 单节点 + 分布式表                           |
| 元数据管理     | 由 **FE 负责管理元数据和查询规划**    | 每个节点本地管理                            |
| 元数据一致性   | 多个 FE 通过 **Paxos 协议**保证一致性 | 依赖 **ZooKeeper / Keeper** 协调            |
| 数据存储与计算 | 由 **BE 负责数据存储和查询执行**      | 每个节点既存储数据又执行查询                |
| 集群扩展       | 原生支持集群，架构天然分布式          | 需要配置 **Distributed Table** 才能形成集群 |
| 运维复杂度     | 集群管理简单，支持自动均衡和故障恢复  | 小规模部署简单，但大规模集群运维较复杂      |
| 适用场景       | 更适合 **大规模分布式数仓场景**       | 更适合 **单机或中小规模分析场景**           |

Doris 采用 FE/BE 分离的原生分布式架构，在集群管理和扩展性方面更加友好；而 ClickHouse 更偏向单机高性能设计，在小规模部署时更加轻量，但在大规模集群环境下需要额外的组件和配置来完成协调与管理。

假设，你在管理一个100节点的集群：

1. 使用Doris，你只需通过FE节点添加新的BE节点，系统会自动处理数据分布；

2. 使用ClickHouse，你需要面对本地表、分布式表、ZooKeeper配置等繁琐操作。

### （二）性能对比

| 对比维度       | Doris                                                        | ClickHouse                                             |
| -------------- | ------------------------------------------------------------ | ------------------------------------------------------ |
| 查询性能       | 在复杂 SQL 分析场景表现优秀，依赖 **MPP 执行框架 + 向量化执行引擎** | 单表查询性能极强，适合极致性能的分析场景               |
| 多表 Join 能力 | 支持 **Shuffle Join**，可以处理大表之间的 Join               | Join 能力相对较弱，大表 Join 场景支持有限              |
| 复杂分析查询   | 更适合复杂 OLAP 查询，如多维分析、复杂聚合                   | 更适合简单查询或单表聚合分析                           |
| 数据更新方式   | 支持 **同步更新与删除**，数据变更后立即可见                  | 更新和删除为 **异步操作**，需要等待后台 Merge 才会生效 |
| 数据一致性     | 更新删除后 **实时可见**，一致性更好                          | 删除后短时间内仍可能查询到旧数据                       |
| 主键模型       | **UniqueKey + Merge-on-Write**，支持真正的主键去重           | 主键去重能力较弱                                       |
| 实时数据场景   | 更适合 **实时看板、用户标签、实时分析等场景**                | 在实时更新频繁的场景中可能出现数据延迟问题             |
| 业务影响       | 数据删除或更新后立即反映在查询结果中                         | 可能出现“数据已删除但查询仍存在”的情况                 |

ClickHouse 在单表查询和极致性能场景下表现突出，而 Doris 依托 MPP 架构和向量化执行，在复杂 SQL 和多表 Join 分析中更具优势。同时，在数据更新和删除方面，Doris 通过同步更新机制保证数据实时可见，更适合实时分析类业务场景。

### （三）生态和易用性

| 对比维度     | Doris                                                        | ClickHouse                                       |
| ------------ | ------------------------------------------------------------ | ------------------------------------------------ |
| 开箱即用     | 开箱即用能力强，部署后即可使用完整 OLAP 功能                 | 基础能力强，但部分功能需要额外配置               |
| SQL 兼容性   | 支持 **标准 SQL**，并兼容 **MySQL 协议**                     | 使用 **自定义 SQL 方言**                         |
| BI 工具集成  | 可直接对接常见 BI 工具（如 Tableau、Superset 等）            | 需要适配或使用专用连接器                         |
| 数据源连接   | 提供丰富 **Connector**，支持对接 **Hive、Iceberg、Hudi、Paimon** 等数据湖 | 数据集成能力较弱，需要额外工具或配置             |
| SQL 功能支持 | 支持较完整的标准 SQL 功能                                    | 不支持部分标准 SQL，如 **相关子查询、EXISTS 等** |
| 学习成本     | 兼容 MySQL 生态，团队上手成本较低                            | SQL 语法差异较大，学习成本相对更高               |
| 社区生态     | 社区活跃，生态发展迅速                                       | 社区成熟，但生态偏向技术型用户                   |

Doris 在易用性和生态集成方面表现更加友好，尤其是对已经拥有 MySQL 技术栈的团队来说，可以快速上手并无缝接入现有 BI 工具和数据湖体系；而 ClickHouse 虽然功能强大，但由于使用自定义 SQL 方言，在学习成本和生态集成方面相对更高。

### （四）使用场景

| 适用场景维度 | Doris                                               | ClickHouse                                  |
| ------------ | --------------------------------------------------- | ------------------------------------------- |
| 查询特点     | 更适合 **复杂 SQL 分析**，如多表 Join、复杂聚合查询 | 更适合 **极致查询性能场景**，尤其是单表分析 |
| 数据更新频率 | 支持 **实时更新与删除**，适合数据频繁变化的业务     | 更适合 **更新不频繁的场景**                 |
| 数据模型     | 支持多种数据模型，灵活性更高                        | 通常以 **宽表模型** 为主                    |
| 运维难度     | 运维简单，适合希望 **易维护、易使用** 的团队        | 通常需要 **专业数据库工程师维护**           |
| 数据集成能力 | 支持与 **多数据源集成**（Hive、Iceberg、Hudi 等）   | 数据集成能力相对较弱                        |
| 并发能力     | 支持 **高并发查询**，适合 BI 与实时分析场景         | 并发能力较好，但更偏向高性能分析            |
| 典型业务场景 | 实时数仓、实时看板、用户标签分析                    | 日志分析、监控分析、行为数据分析            |

ClickHouse 更适合对查询性能要求极高、数据模型稳定且更新较少的分析场景；而 Doris 则在复杂 SQL 支持、实时数据更新以及系统易用性方面更具优势，更适合作为企业级实时数仓和多维分析平台。

> 如果你的团队资源有限，追求**易用性和全面性，Doris**是更好的选择；如果你只追求**极致的单表查询性能，且有专业团队维护，ClickHouse**可能更适合。



## 二、快手ck转doris的实践

快手将OLAP系统从原有的“湖仓分离”架构升级为基于Apache Doris的**“湖仓一体”**架构，以解决旧架构面临的核心挑战，并介绍了新架构中的关键服务和优化实践。

### （1）原有架构的问题

快手原有的OLAP系统采用湖仓分离架构，离线数据湖核心为Hive/Hudi，实时数仓核心为ClickHouse。这套架构随着业务发展，暴露出以下主要问题：
- **存储冗余**：数据需导入ClickHouse以提高查询性能，导致数据在湖和仓中重复存储。
- **资源占用与抢占**：数据导入和ClickHouse内部的Compaction等过程会占用大量集群资源，在高并发查询时，读写资源竞争激烈，影响整体性能。
-  **治理复杂**：数据工程师需投入大量人力构建和维护ADS层模型及导入任务，下线看板对应的冗余任务清理依赖人工，运维管理复杂度高。
- **查询调优难**：ClickHouse的排序字段、二级索引等配置对查询性能影响大，但学习及操作门槛高，调优难度大。

### （2）**升级目标与选型**

为解决上述问题，快手希望引入湖仓一体架构，让数仓能直接分析湖中的数据，避免复杂的数据传输。他们选择了**Apache Doris**，因为Doris在湖仓一体方向已有诸多探索，具备深度适配湖格式、多源联邦分析、湖仓数据无缝集成和统一数据写入与计算的能力。

**基于Apache Doris的新湖仓一体架构**

新架构主要由数据加工层、数据缓存层和数据查询层构成，并引入了两个核心服务：

- **数据加工层**：在数据湖仓（Hive/Hudi）中完成从ODS到DWS层的处理。

- **数据缓存层**：使用Alluxio缓存ADS层数据，并配套**元数据缓存服务（Meta Server）**，实现查询引擎、元数据服务与缓存服务的三方联动，显著降低元数据访问延迟（从平均800毫秒降至约50毫秒）。

- **数据查询层**：使用**Apache Doris替换ClickHouse**，直接访问湖仓数据提供高性能查询。

- **查询路由服务**：将扫描量过大的查询自动路由到Spark引擎，避免占用过多Doris资源。

- **自动物化服务**：实现DWS到ADS层的按需自动化加工与智能查询加速。

### （三）**核心收益**

通过引入Apache Doris，快手获得了以下核心收益：

1. **统一存储，简化链路**：Doris可直接分析湖中数据，消除了向OLAP系统导入数据的需求，降低了存储与维护成本，缩短了数据链路。

2. **查询性能提升**：凭借Doris自身的高性能计算引擎、物化视图透明改写和基于代价的优化器，满足了不同场景的查询加速需求。

3. **更灵活的数据治理**：结合Doris的物化视图改写能力和自研的自动物化服务，简化了在ADS层的视图建模，对业务层屏蔽了底层复杂性。

通过引入Apache Doris的湖仓一体能力，快手成功解决了湖仓分离架构的痛点，实现了存储、链路的简化与查询性能的提升。
