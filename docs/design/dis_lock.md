# 分布式锁

> 面试和面试之外

## 面试常问

#### 分布式锁是什么

分布式锁是一种在分布式系统中协调多个进程或者线程对共享资源进行互斥访问的机制。它能够确保在任意时刻只有一个节点能够获得锁并执行关键代码，从而避免数据的不一致和资源竞争。-

---



#### 一个合格的分布式锁的要求

1. 能够保证在分布式部署的应用里面，同一个方法在一个时刻只能被一台机器上的一个线程执行。
2. 这把锁可重入。
3. 获取锁和释放锁的性能要高。
4. 有时候锁需要有阻塞的功能。

---



#### 分布式锁有几种实现方式？

1. 基于数据库实现
   
2. 基于缓存实现分布式锁。
3. 基于zookeeper实现分布式锁。

---



#### 如何基于数据库实现分布式锁

##### 1. 基于数据库的唯一索引

创建一张锁表，然后操作表里面的数据来实现。如果需要锁住某个方法或者资源，就在表里面增加一条记录，释放锁的时候就删除这条记录。

```SQL
CREATE TABLE `distributed_lock` (
    `lock_key` VARCHAR(64) NOT NULL,
    `lock_value` VARCHAR(128) NOT NULL,   -- 用于标识持有者，如 UUID
    `expire_time` TIMESTAMP NOT NULL,      -- 锁超时时间
    `create_time` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (`lock_key`),
    UNIQUE KEY `uniq_lock_key` (`lock_key`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

- 获取锁：尝试插入一条记录。如果插入成功表示获取锁，否则表示锁被其他客户端所持有。
- 释放锁：执行完业务之后，删除key对应的记录。为了防止死锁，需要在业务完成后或者发生异常的时候释放锁。
- 关于**锁超时**处理，需要借助定时任务扫描，或者在其他客户端在尝试获取锁的时候，删除并重试。

缺点：

1. 性能受到数据库的限制，不适合高并发场景。
2. 依赖数据库的可用性，存在单点故障。
3. 没有自动释放的机制，需要额外处理超时。
4. 这把锁是非阻塞的。

优点：

- 实现简单，不依赖第三方的中间。
- 数据天然持久化。

##### 2. 基于悲观锁

原理是利用数据库的行级锁，在事务中通过`select……for update`锁定特定的行，实现互斥。

```SQL
CREATE TABLE `lock_resource` (
    `id` INT PRIMARY KEY AUTO_INCREMENT,
    `resource_name` VARCHAR(64) UNIQUE NOT NULL,
    `locked` TINYINT DEFAULT 0  -- 可选，用于标识
) ENGINE=InnoDB;
```

创建锁表，预先插入代表资源的锁记录。

- 获取锁：开启事务，执行select for update，如果查询成功，表示获得锁，否则回滚事务。 for update会锁定这一行，直到事务提交或者回滚。

- 释放锁：提交事务，或者在finally中回滚事务。

优点：

- 利用数据库内置的锁机制，简单可靠。
- 锁自动释放。

缺点：

- 长事务会导致数据库连接的占用，影响并发。
- 数据库单点，需要高可用。
- 不适合分布式系统跨数据库实例的场景。

##### 3. 基于乐观锁

通过版本号控制并发更新，适用于冲突概率低的场景。

实现：在数据库中增加一个版本号字段version，获取锁的时候读取当前的版本号，更新数据时候使用cas。如果影响行数为1，表示获取成功，否则失败重试。

缺点：

- 不适合写冲突高的场景，会导致大量重试。
- 没有自动阻塞，需要应用层自旋。

#### 如何用redis实现分布式锁

利用redis单线程特性，在多个redis客户端同时通过`setnx`命令尝试获取锁，如果返回1表示获取成功，否则表示获取失败。由于rediss是单线程，所以可以保证只会有一个客户端成功获取，其他客户端会失败。

- 获取成功之后可以设置一个过期时间，防止客户端挂了之后一直持有这个锁。
- 客户端在释放锁的时候，需要先判断这个锁是否属于这个客户端，如果是的话，用删除命令释放锁。

>  Redis的setnx命令，在指定的key不存在的时候，为key设置指定的值返回1，设置失败返回0。

优点：

- 命令实现简单易于理解和使用。
- 性能较高，由于命令执行原子性，保证分布式锁的正确性，又因为redis里面是单线程执行，所以性能较高。

缺点：

- 锁无法续期，可能会导致锁误释放。
- 如果客户端崩溃，可能会导致死锁。
- 不支持重入。

> [!TIP]
> 可以用redisson，这是一个更加完善的分布式锁。
