# 分布式锁

> 面试和面试之外

## 面试常问

#### 分布式锁是什么

分布式锁是一种在分布式系统中协调多个进程或者线程对共享资源进行互斥访问的机制。它能够确保在任意时刻只有一个节点能够获得锁并执行关键代码，从而避免数据的不一致和资源竞争。-

---



#### 一个合格的分布式锁的要求

1. 能够保证在分布式部署的应用里面，同一个方法在一个时刻只能被一台机器上的一个线程执行。
2. 这把锁可重入。
3. 获取锁和释放锁的性能要高。
4. 有时候锁需要有阻塞的功能。

---



#### 分布式锁有几种实现方式？

1. 基于数据库实现
   
2. 基于缓存实现分布式锁。
3. 基于zookeeper实现分布式锁。

#### 如何基于数据库实现分布式锁

##### 1. 基于数据库的唯一索引

创建一张锁表，然后操作表里面的数据来实现。如果需要锁住某个方法或者资源，就在表里面增加一条记录，释放锁的时候就删除这条记录。

```SQL
CREATE TABLE `distributed_lock` (
    `lock_key` VARCHAR(64) NOT NULL,
    `lock_value` VARCHAR(128) NOT NULL,   -- 用于标识持有者，如 UUID
    `expire_time` TIMESTAMP NOT NULL,      -- 锁超时时间
    `create_time` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (`lock_key`),
    UNIQUE KEY `uniq_lock_key` (`lock_key`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

- 获取锁：尝试插入一条记录。如果插入成功表示获取锁，否则表示锁被其他客户端所持有。
- 释放锁：执行完业务之后，删除key对应的记录。为了防止死锁，需要在业务完成后或者发生异常的时候释放锁。
- 关于**锁超时**处理，需要借助定时任务扫描，或者在其他客户端在尝试获取锁的时候，删除并重试。

缺点：

1. 性能受到数据库的限制，不适合高并发场景。
2. 依赖数据库的可用性，存在单点故障。
3. 没有自动释放的机制，需要额外处理超时。
4. 这把锁是非阻塞的。

优点：

- 实现简单，不依赖第三方的中间。
- 数据天然持久化。

##### 2. 基于悲观锁

原理是利用数据库的行级锁，在事务中通过`select……for update`锁定特定的行，实现互斥。

```SQL
CREATE TABLE `lock_resource` (
    `id` INT PRIMARY KEY AUTO_INCREMENT,
    `resource_name` VARCHAR(64) UNIQUE NOT NULL,
    `locked` TINYINT DEFAULT 0  -- 可选，用于标识
) ENGINE=InnoDB;
```

创建锁表，预先插入代表资源的锁记录。

- 获取锁：开启事务，执行select for update，如果查询成功，表示获得锁，否则回滚事务。 for update会锁定这一行，直到事务提交或者回滚。

- 释放锁：提交事务，或者在finally中回滚事务。

优点：

- 利用数据库内置的锁机制，简单可靠。
- 锁自动释放。

缺点：

- 长事务会导致数据库连接的占用，影响并发。
- 数据库单点，需要高可用。
- 不适合分布式系统跨数据库实例的场景。

##### 3. 基于乐观锁

通过版本号控制并发更新，适用于冲突概率低的场景。

实现：在数据库中增加一个版本号字段version，获取锁的时候读取当前的版本号，更新数据时候使用cas。如果影响行数为1，表示获取成功，否则失败重试。

缺点：

- 不适合写冲突高的场景，会导致大量重试。
- 没有自动阻塞，需要应用层自旋。

#### 如何用redis实现分布式锁

利用redis单线程特性，在多个redis客户端同时通过`setnx`命令尝试获取锁，如果返回1表示获取成功，否则表示获取失败。由于rediss是单线程，所以可以保证只会有一个客户端成功获取，其他客户端会失败。

```LUA
SET key value NX PX [milliseconds]
```


> Redis的setnx命令，在指定的key不存在的时候，为key设置指定的值返回1，设置失败返回0。

1. **加锁**：尝试向 Redis 写入一个键值对，键为锁的唯一标识（如资源名），值为客户端唯一标识（如 UUID）。使用 `SET key value NX PX milliseconds` 命令，该命令是原子操作：
   - `NX`：只有键不存在时才设置成功，保证互斥性。
   - `PX`：设置键的过期时间（毫秒），防止客户端崩溃导致死锁。
2. **解锁**：通过 Lua 脚本或事务，先验证当前锁的值是否为自己的标识，只有匹配时才删除键，避免误删其他客户端的锁。
3. **可重入**：如果同一个客户端需要多次获取同一把锁，可以借助 ThreadLocal 或 Redis 的 hash 结构记录重入次数（如 Redisson 的实现）。

**为什么用 Lua 脚本？**：因为解锁操作需要“先取值比较，再删除”，这两个步骤必须原子执行，避免在比较之后删除之前锁被其他客户端篡改。

**注意**：不能用 `jedis.del(key)` 直接删除，否则可能误删其他客户端的锁。

优点：

- 命令实现简单易于理解和使用。
- 性能较高，由于命令执行原子性，保证分布式锁的正确性，又因为redis里面是单线程执行，所以性能较高。

缺点：

- 锁无法续期，可能会导致锁误释放。
- 如果客户端崩溃，可能会导致死锁。
- 不支持重入。

> [!TIP]
> 可以用redisson，这是一个更加完善的分布式锁。

#### 如何使用zookeeper实现分布式锁

> 核心是基于zk的临时有序节点，实现分布式锁

大致思想是，每个客户端对某个方法加锁的时候，在zookeeper上的与该方法对应的指定节点的目录下，生成一个唯一的**瞬时有序节点**。

判断是否获取锁：只需要判断有序节点中序号最小的一个。

释放的时候：需要将这个瞬时节点删除，同时可以避免服务宕机导致锁无法释放产生的死锁问题。

##### 能解决的问题

锁的释放：使用zookeeper可以解决锁无法释放的问题，因为创建的时候是在zk里面建一个临时节点，一旦客户端突然挂掉之后，session连接会断开。那么这个临时节点会自动删掉。

阻塞：使用zk可以实现阻塞锁，客户端可以在zk创建节点，并且在节点绑定监听器，一旦节点变化，就会通知客户端。客户端可以检查自己的节点是不是所有节点中最小的，如果是的话就可以获取到锁，可以执行。

可重入性： zk可以有效解决不可重入的问题，客户端在创建节点的时候，把当前客户端的主机跟线程直接写到节点，下次想获取的时候直接比对就可以。

单点问题：使用zk的时候是集群部署，只要集群中半数以上机器存活就可以对外提供服务。

### RedLock是什么

 RedLock是redis的作者提出的一个多节点分布式锁算法，用来解决单节点的分布式锁可能存在的问题。

单点故障问题：

- 如果单个实例挂掉，那么使用这个实例的客户端无法获取锁。
- 如果用集群模式部署，如果一个master加锁成功，但是没有同步就挂了，这个时候的新master会认为这个锁不存在。

 Redlock通过使用多个redis节点来提供一个更加健壮的分布式锁解决方案，能够在某些redis节点故障的情况下，仍然能够保证分布式锁的可用性。

步骤：

1. 客户端在想要锁的时候会生成一个全局唯一的id，官方建议用系统时间来生成。
2. 客户端尝试用这个id获取所有redis节点的同意，这一步会使用setnx命令来实现。
3. 如果半数以上的节点同意，那这个锁就成功设置。

大佬争论：

但是两个大佬针对redlock有过讨论，马丁认为红锁存在潜在的问题，比如网络延迟，时钟漂移等问题。他认为使用redis的set nx ex更加简单可靠安全。

但是redis作者认为这个是安全的，并且可以保证分布式锁的正确性。他认为这些问题都可以在一定程度上得到解决。

- 网络分区：网络分区的情况下，红色仍然可以提供可靠性，虽然可能会存在节点获取相同的锁，但这个情况只会发生在网络分区，并且只会发生在一小部分节点，在网络分区恢复之后，红锁会自动解锁。
- 时间漂移：可以使用ntp等工具来同步不同机器时间。
-  Redis主从复制可能导致锁的丢失，但可以通过其他方式避免，比如redis的cluster。

### 分布式锁需考虑问题

#### 锁的基本要求

一个分布式锁需要考虑**互斥性**，**可重入性**和**锁的性能**。

- 对于锁的互斥性，可以借助setnx来解决。这个操作是原子操作，并且结合单线程，可以保证互斥性。
- 由于redis基于内存，所以性能也很高。
- 至于可重入性，就是在锁没有释放的情况下，可以反复拿到同一把锁，这个可以通过在锁里记录加锁次数，用setnx也可以做到。



#### 锁的误解除

核心是确保锁的持有者才能释放锁，避免其他客户端错误的解除锁。

为了解决这个问题，可以在加锁的时候，把锁的唯一信息放进去，然后做一次判断。这个可能需要借助lua脚本。



#### 锁的有效时间

为了避免死锁，一般会给分布式锁设置一个超时时间。

但如果代码执行比较慢的话，会导致锁会自动解除，那么其他的线程就可以拿到锁，这个时候就会产生并发的问题。

所以需要一个办法来解决，一种是设置一个很长的超时时间，但这个会影响并发，比如解锁失败。还有一个办法就是用一个watch dog机制，给锁自动续期，让他不会提前释放。



#### 锁的单点故障问题

使用单节点做分布式锁的话，如果这个实例挂掉，那么所有这个实例的客户端都会无法获取锁。为了解决这个问题，可以引入集群模式。通过哨兵来检测redis实例挂掉的情况，提升集群的可用性。

但这也会有一个问题：如果用集群模式部署， Master在一个客户端的节点加锁成功，但是没有及时的同步到其他节点就挂了，新的客户端来加锁的时候，新master会认为这个锁不存在。

为了解决这个问题，作者提出了**redlock**。



#### 网络分区问题

引入红锁之后会存在网络分区问题。在网络分区的情况下，如果集群发生了脑裂，不同的节点可能会获取到相同的锁，也会导致分布式的不一致。

但是这个问题出现的概率比较小，受影响的也只是一部分的节点。



#### 时间漂移问题

由于不同的机器之间，时间可能存在微小的偏移，可能会导致锁的失效时间不一致，也会导致分布式的不一致问题。

就会导致有的redis实例已经解锁，会使新的客户端可以拿到锁。这个问题的**解决方案**就是红锁可以用ntp等工具来同步不同机器之间的时间，从而避免这个问。
