# cpu&amp;指令重排

## cpu是什么

CPU 是中央处理器的简称，是计算机的大脑，主要任务是执行机器指令、处理数据以及控制其他硬件设备。

CPU 接受来自内存或者输入设备的指令，然后按照指令执行运算和逻辑处理，最终输出结果。

对于 CPU 来说，只认识 0 和 1 的二进制序列。对于人来说，二进制序列的可读性太差，于是发明了指令。指令就是把机器码中特定的 01 排序简化成对应的指令，一般是英文简写，比如 mov 和 inc，可读性稍好。

CPU 类型根据处理数据所使用的芯片类型来指定，生成芯片的制造商主要有英特尔、AMD、ARM。对于这些公司，它们实际上并不会生产设备，但是会将有价值的高端处理器设计和其他专有技术出租给其他生产设备的公司，比如苹果公司。

## 指令集

不同的制造商生产的 CPU 执行同一个操作，对应的机器码可能不同，对应的指令也会有差别。因此每个类型的 CPU 所支持的指令集合，称之为对应类型的 CPU 指令集，即不同的 CPU 使用不同的指令集。

CPU 和指令集不同导致相关产业生态不同，目前个人计算机和服务产品以 X86 的 CPU 为主，移动端和部分嵌入式系统以 ARM 的 CPU 为主。

由于指令集的可读性比较差，于是又发明了汇编语言。在汇编语言中，用助记符代替机器指令，用地址符号或者标号代替指令或操作数的地址。

在不同的硬件平台，汇编语言对应不同的机器语言指令集，通过汇编过程转换成机器指令。

为了使计算机用户编程更简单，后面出现了各种高级计算机语言。高级语言比机器语言、汇编语言更接近人类语言。当计算机执行高级语言编写的程序时，仍然需要把程序解释和编译成机器码的指令码。完成这个过程的程序，就叫编译程序或者解释程序。

## cpu的组成以及工作流程

### 组成

CPU 通常会被看成一个整体，用来接收指令、返回执行的结果，感觉是一个比较简单的流程。但如果深入 CPU 的内部，会发现 CPU 是由多个功能构件组成，大致可以分为控制模块、运算模块和存储模块。这三部分是由 CPU 的总线连接起来的。

- **控制单元**是计算机的指挥中心，只有在它的控制下，整个 CPU 才能有条不紊地工作、自动执行程序。控制单元包括指令寄存器、指令计数器，其中指令寄存器存放当前正在执行的指令，指令计数器总是指向下一条要执行指令的地址。

- **运算单元**由算术单元和逻辑单元组成，可以对两个输入值执行算术或者逻辑运算，并产生输出值。

- **存储单元**也叫寄存器，是 CPU 内部的高速存储单元，用于临时存放指令、数据和地址。寄存器因为其极高的数据访问速度，对提升 CPU 效率有重要的作用。

### 工作流程

CPU 从 cache 或者主存中取出指令，然后放入指令寄存器。指令译码器对该指令进行译码，最终把指令分解为一系列的微操作，然后发出各种控制命令，从而完成一条指令的执行。

一条指令的执行大概分为 5 个阶段：读取指令、译码、执行、访问内存、写回。每个阶段都有一个功能部件进行处理。

虽然每条指令是顺序执行，但如果指令之间没有依赖关系，后一条指令并不需要等到前一条指令完全执行完成再开始执行，而是前一条指令完成取指之后，后一条指令便可以开始执行取指操作，从而达到并行执行的效果。

#### 乱序执行

为了提高 CPU 的执行效率，CPU 采用了乱序执行。

首先乱序执行是有前提的：
1. 有数据依赖的两个操作是不能重排序的
2. 要遵循 as-if-serial：重排后的结果与程序代码本身在单线程下的执行结果应该是一样的

只要不影响单线程程序执行的结果，不违背上面的两个条件，就可以对两个指令重排序。

乱序执行技术是处理器为提高运算速度，而作出违背代码原有顺序的优化。通过乱序执行，合理调整指令的执行顺序，可以提高流水线的运行效率，让指令的执行能够尽可能地并行起来。

编译器重排序优化的目的是利用 CPU 的乱序执行。与处理器乱序执行相比，编译器重排序能够完成更大范围、效率更好的乱序优化。

**有序性**指的是程序执行的顺序按照代码的先后顺序执行。

**指令重排序**指的是编译器和处理器为了优化程序性能，在不改变单线程程序执行结果的前提下，对指令的执行顺序进行调整。

在单线程的环境中，有序性是可以保证的。但是在多线程环境中，由于存在指令的重排序，如果不加以控制，指令重排可能会导致程序执行结果不一致，从而引发并发错误。

有序性是为了保证程序在并发时的正确，防止因指令重排导致错误。

### 重排导致问题的例子

举一个例子：单例模式。

`instance = new Singleton()` 这个并不是原子操作，会分为三步：
1. 在内存中申请一块空间，new 出一个对象
2. 调用 new 出的这个对象的构造方法
3. 将这个对象的引用赋值给 instance

这个是我们期望的执行顺序，但是计算机为了提高运行效率，会将我们的指令顺序进行优化重排，比如优化成 1→3→2。

单线程的情况下不会出现问题，但是多线程的情况下就可能会出问题。

#### 解决

Java 的关键字 `synchronized` 可以保证原子性、可见性和有序性。但是这里的有序性并不代表能够禁止指令重排序，它保证有序性的本质是将并发变成串行，谁获得锁谁执行。但不能保证内部指令重排序的问题，所以 `synchronized` 不能禁止指令的重排序。

所以如果不考虑效率的情况下，想让 CPU 按照程序的顺序执行，不让 CPU 乱序，只能禁止编译器重排序优化，阻止 CPU 的乱序执行。这里 JVM 提供了 `volatile` 关键字。

## cpu和缓存

CPU 的处理能力远比内存强，内存一次的读写操作，可以够处理器执行上百条指令。为了弥补处理器与内存处理能力之间的差距，在内存和处理器之间引入了高速缓存。

高速缓存的容量比内存要小，但是交换速度比内存要快得多。每个处理器都有自己的高速缓存，一般直接跟 CPU 芯片集成或位于主板总线互联的独立芯片上。

一般的缓存都是三级架构：L1、L2、L3。

CPU 的读操作实质上就是从缓存中读取数据到寄存器。在多级缓存的架构中，如果缓存找不到数据，就会层层读取缓存；当所有的缓存都找不到对应数据，就会去内存里面寻址。寻址到的数据首先放在寄存器，其次会在 CPU 的缓存中保存一份副本的数据。

CPU 的写也是针对缓存操作的，并不会直接和内存打交道，而是通过某种机制实现数据从缓存到内存的更新。

当 CPU 修改缓存中的数据之后，这些修改什么时候能够传播到内存？

有两种方案：
1. **写直达**：缓存中的任何一个字节的修改都会立刻传播到内存，这种方法像是穿透内存，所以用 through 来命名。
2. **写回**：当 CPU 采取写回策略时，对缓存的修改不会立即传播到内存，只有当缓存块被替换的时候，这些被修改的缓存块才会写回并覆盖到内存中的过时数据。一般都是采用延迟写回的策略。

如果采用延迟写回的策略，在多线程多核处理器情况下，需要一种机制来同步不同处理器里面的缓存数据。

有两种方案：
1. **写传播**：写传播解决的是感知问题，如果一个核心修改数据，就需要同步给其他的核心。但是这种做法会导致一个问题，如果各个核心收到的同步信号顺序不一致，那最终的同步结果也会不一致。
2. **事务的串行化**：也就是加锁，哪个核心拿到锁才能够对对应的数据进行处理。

### 写传播和事务串行化的实现

写传播的实现方式是**总线嗅探**。

总线除了能在一个主模块和一个从模块之间传输数据，还支持一个主模块对多个从模块写入数据，这种操作就是广播。

要实现写传播，其实就是将所有的写操作广播到所有的 CPU 核心，而其他 CPU 核心时刻监听总线上的广播，再修改本地的数据。

事务串行化的实现是**总线仲裁**。

总线的独占性要求同一时刻最多只有一个主模块占用总线，天然地将所有核心对内存的读写操作串行化。如果多个核心同时发起总线事务，总线仲裁单元会对竞争作出仲裁，未获胜的事务只能等待获胜的事务处理完之后才能执行。

### 缓存副本的处理策略

其他 CPU 接收到某个 CPU 的修改缓存通知后，该如何处理自身内部缓存中所保存的相应数据副本？

有两种方案：

1. **写更新**：某个 CPU 缓存更新时，该 CPU 必须发起一次总线请求，通知其他 CPU 将他们的缓存更新为刚写入的指令。所以这种方式会占用总线的带宽。

2. **写无效**：某个 CPU 缓存更新的时候，这个 CPU 会发出总线请求，通知其他的 CPU 缓存无效，用到的时候重新从内存或者 CPU 中获取。这个方式和现在 JMM 的设计很像。

在具体的实践中，绝大多数 CPU 都会采用**写无效**的策略。这是因为多次写操作只需要发起一次总线事件即可，第 1 次写已经将其他缓存的值设为无效，之后的写不必再更新状态。这样可以有效的节省 CPU 核间的总线带宽。

简而言之就是：一次设置之后，接下来的写操作不用设置。

### 缓存行

CPU 的 cache 和内存系统用固定大小的数据块进行交互，这个数据块被称为**缓存行**。

CPU 从内存里面读取数据到缓存行的时候，并不是只读自己想要的部分，而是读取足够的字节填入缓存行。如果 CPU 访问相邻的数据时，就不必每次从内存读取而提高速度。

这种缓存行的依据是**局部性原理**，又分为：
- **空间局部性**：相邻的数据可能会被访问
- **时间局部性**：刚访问过的数据可能很快会被再次访问

### 事务串行化的解决方案

一般情况下有两种方式，一种是在总线加锁，一种是通过缓存一致性协议，这两种方案都是硬件来提供的。

1. **总线加锁**：早期的 CPU 中是通过 CPU 上加上锁的形式来解决缓存不一致的问题。因为 CPU 与其他部件进行通信是通过总线来进行，如果对总线加锁的话，会阻塞其他 CPU 对其他部件的访问（比如内存），从而使得只有一个 CPU 能够访问内存。由于在锁住总线期间，其他 CPU 无法访问内存，所以导致效率低下。

2. **缓存一致性协议**：由于总线加锁效率比较低，所以出现缓存一致性协议。缓存锁的粒度较小，锁的是缓存行。核心思想是：当 CPU 对某个缓存行中的数据进行写操作之后，通知其他 CPU 放弃存储在他们内部的缓存，或者从主内存里面重新读取。

### MESI 协议

MESI 协议基于总线嗅探机制实现事务的串行化，引用状态机机制降低了总线的带宽压力。

主要是通过 4 种状态之间的切换，保持同一数据在不同 CPU 的一致性：

- **M (Modified)**：本地处理器已经修改缓存行，它的内容与内存中的内容不一样，并且缓存只有本地的一个拷贝
- **E (Exclusive)**：缓存行内容和内存中的一样，而且其他处理器都没有这行数据
- **S (Shared)**：缓存行内容和内存中的一样，有其他处理器也存在这个缓存行的拷贝
- **I (Invalid)**：缓存行失效，不能使用

#### 读取数据的流程

当 CPU 读取数据的时候：

- 如果缓存行的状态是 **M** 或 **S**，可以直接用缓存的数据
- 如果缓存行的状态是 **I** 或者不存在，会发出信号等待其他 CPU 的监听结果
  - 如果其他 CPU 没有缓存这个数据，就直接从内存里面拿并且标记为 **E**
  - 如果其他 CPU 有这个数据且状态是 **M**，则需要等待把缓存更新到内存之后再读取
  - 如果其他 CPU 有这个数据的缓存且状态为 **E** 或者 **S**，就从内存里面去读取，并把自己的状态设置为 **S**

#### 写入数据的流程

当 CPU 写数据的时候：

- 如果发现缓存行是 **M** 或者 **E**，就可以直接执行
- 如果发现缓存行是 **S**，代表此时其他的 CPU 也存在这个变量的副本，就会发出信号通知其他 CPU 将变量的缓存行设置为无效的状态，这个时候会有比较大的性能开销
- 在写入完成之后把它的缓存行设置为 **M**
- 当其他 CPU 需要读取这个变量，发现自己的缓存行是无效的时候，就会从内存重新读取
