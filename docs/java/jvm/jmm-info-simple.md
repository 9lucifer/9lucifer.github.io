# java内存模型概述

## 背景

并发编程面临的三个核心问题：

1. **原子性问题**
   - 操作系统分为进程和线程，**分时复用** CPU
   - CPU 会在不同的线程之间切换提高效率，切换的时机是 CPU 执行完一条指令
   - 高级语言翻译成机器语言的时候可能会有多条，这个时候可能就会出现原子性问题

2. **有序性问题**
   - 为了提高 CPU 的运算效率，CPU 可能会对输入的指令进行乱序执行，也就是所谓的指令重排
   - 在单核 CPU 的情况下会提升运行的效率
   - 在多线程的情况下会出现一些问题，被称为有序性问题

3. **可见性问题**
   - CPU 由单核发展为多核，同时为了均衡与内存速度的差异，CPU 增加了缓存
   - 但是多核之间会有缓存的一致性问题，这个问题被称为可见性问题
   - 解决可见性问题的方案有总线锁和缓存一致性协议

### 总线锁 vs 缓存锁

**总线锁**：
- 如果对总线加锁，会阻塞其他 CPU 对其他部件的访问，从而使得只有一个 CPU 能够访问内存
- 这种方式效率比较低

**缓存锁**：
- 缓存锁锁的是 CPU 的缓存行，粒度比较小
- 缓存锁有很多的实现方案，最著名的是 MESI 协议
- 但是如果 CPU 完全遵循 MESI 协议，可能会有性能问题

MESI 协议的思想是给缓存行设置状态，如果有其他的 CPU 要来对缓存进行写操作的时候，需要读取状态，并判断是否需要通知其他的 CPU 放弃缓存，或者重新从主内存里面去读取。

这个性能问题就是：当前 CPU 发出通知需要等待其他 CPU 响应，这个等待的时间就是性能的损耗。

### 对缓存锁的优化

为了解决 MESI 协议的性能问题，设计者提出了 **Store Buffer** 和 **Invalid Queue**。

- CPU 在发送通知的时候直接把消息放到 Store Buffer 里面，不再等待其他 CPU 的响应，直接进行下一个操作
- 对于其他的 CPU 来说，当它接收到信息之后，并不是立刻把缓存行标记为失效，而是放到 Invalid Queue 进行一个延迟操作

但是这么做的话，又导致缓存的顺序一致性变成了弱缓存一致性，又引出了可见性问题。

对于这个问题，涉及这个协议的工程师认为这个概率是比较小的，他们提供了一个**内存屏障**的 API，让我们在用得到的地方进行添加。

于是下面就引出了 Java 的内存模型。

Java 定义了一套内存规范：
- 包括主内存与工作内存
- 提供了一系列的原语封装底层实现，供开发者使用
- 提供了 happens-before 规则
- 提供了 volatile 关键字

## JMM 起源

Java 虚拟机规范试图定义一种 Java 内存模型，来屏蔽各种硬件和操作系统的内存访问差异，以实现让 Java 程序在各种平台上都能达到一致的内存访问效果。

在此之前，主流程序如 C 或 C++ 直接使用物理硬件和操作系统的内存模型，会导致程序和硬件平台绑定——某个程序在 A 机器上并发的很好，但是在 B 机器上频繁出错，所以不得不根据平台来编写程序。

Java 的内存模型逐渐成熟起来，它规定了线程如何与主内存和工作内存交互，以确保多线程程序的**可见性、有序性和原子性**。

### 问题根源与解决方案

- **可见性**的原因是 CPU 的缓存
- **有序性**的问题是编译器的优化和 CPU 的乱序执行

解决可见性、有序性问题的极端方案是禁用 CPU 缓存，禁止编译器优化以及 CPU 乱序执行，但是这样性能也会下降。

合理的方案是按照需求禁用缓存和编译优化，在需要的地方添加相应的编码。Java 的内存模型规范了 JVM 如何按需禁用缓存和编译优化。

对于**原子性**问题，这个是因为高级语言翻译成 CPU 指令时，不是 1 对 1 的关系，再加上 CPU 对线程的切换。

解决方案就是在高级语言翻译成 CPU 指令的时候进行打包。Java 也提供了 `synchronized` 关键字来包裹代码块，保证原子性。
