# java内存模型概述

### 背景

1. 操作系统分为进程和线程，**分时复用**cpu；cpu会在不同的线程之间切换提高效率，切换的时机是cpu执行完一条指令。高级语言翻译成机器语言的时候可能会有多条，这个时候可能就会出现**原子性问题**。
2. 为了提高cpu的运算效率，cpu可能会对输入的指令进行乱序执行，也就是所谓的指令重排，在单核cpu的情况下会提升运行的效率。在多线程的情况下会出现一些问题，被称为**有序性问题**。
3.  CPU由单核发展为多核，同时为了均衡与内存速度的差异，cpu增加了缓存。但是多核之间会有缓存的一致性问题，这个问题被称为**可见性问题**。解决可见性问题的方案有总线锁和缓存一致性协议。

总线锁：如果对总线加锁，会阻塞其他cpu对其他部件的访问，从而使得只有一个cpu能够访问内存。这种方式效率比较低。

缓存锁：缓存锁锁的是cpu的缓存行，粒度比较小。缓存锁有很多的实现方案，最著名的是MESI协议。但是如果cpu完全遵循MESI协议，可能会有性能问题。 MESI协议的思想是给缓存行设置状态，如果有其他的cpu要来对缓存进行写操作的时候，需要读取状态。并判断是否需要通知其他的cpu放弃缓存，或者重新从主内存里面去读取。这个性能问题就是，当前cpu发出通知需要等待其他cpu响应，这个等待的时间就是性能的损耗。

对缓存锁的优化：为了解决mesi协议的性能问题，设计者提出了store buffer和invalid queue。 Cpu在发送通知的时候直接把消息给放到store buffer里面，不再等待其他cpu的响应，直接进行下一个操作。对于其他的cpu来说，当他接收到信息之后，它并不是立刻把缓存行标记为失效，而是放到invalid queue进行一个延迟操作。但是这么做的话，它又导致缓存的顺序一致性变成了弱缓存一致性，又引出了可见性问题。对于这个问题，涉及这个协议的工程师认为这个概率是比较小的。他们提供了一个内存屏障的api，让我们在用得到的地方进行添加。

于是下面就引出了Java的内存模型。Java定义了一套内存规范，包括主内存与工作内存；也提供了一系列的原语封装底层实现，供开发者使用；并且提供了happens-before规则，还有volatile关键字。

#### JMM起源

 Java虚拟机规范试图定义一种Java内存模型，来屏蔽各种硬件和操作系统的内存访问差异，以实现让Java程序在各种平台上都能达到一致的内存访问效果。

在此之前，主流程序如c或c++直接使用物理硬件和操作系统的内存模型，会导致程序和硬件平台绑定，某个程序在a机器上并发的很好，但是在b机器上频繁出错。所以不得不根据平台来编写程序。

java的内存模型逐渐成熟起来，它规定了线程如何与主内存和工作内存交互，以确保多线程程序的可见性，有序性和原子性。

导致可见性的原因是cpu的缓存，导致有序性的问题是编译器的优化和cpu的乱序执行。那么我解决可见性有序性问题就是禁用cpu缓存，禁止编译器优化以及cpu乱序执行。但是这样性能也会下降。合理的方案是按照需求禁用缓存和编译优化，在需要的地方添加相应的编码。 Java的内存模型规范了jvm如何按需禁用缓存和编译优化。
对于原子性问题，这个是因为高级语言翻译成cpu指令时，不是1对1的关系，再加上cpu对线程的切换。这个的解决方案就是在高级语言翻译成cpu指令的时候进行打包。 Java也提供了synchronized的关键字来包裹代码块，保证原子性。
