# mysql数据库——锁

## 一、什么是数据库锁

在数据库系统中，尤其是**c端**场景，通常会有多个事务同时访问同一份数据的情况；比如在电商系统里面，会存在多个用户同时购买同一件商品，这个时候多个事务就会并发读写和修改库存。

如果没有控制机制，就可能会出现数据不一致的问题，比如：

<img src="https://imgtu.oss-cn-beijing.aliyuncs.com/blog_img/image-20260314000315667.png" alt="image-20260314000315667" style="zoom:50%;" />

最终库存变成 **99**，而正确结果应该是 **98**。这种情况就是典型的 **并发更新问题**。

除了并发更新问题，在数据库并发访问过程中，还可能出现以下几种典型问题：
1. 脏读

脏读是指 **一个事务读取到了另一个事务尚未提交的数据**。

例如：

```
事务A：修改余额 100 → 50（未提交）
事务B：读取余额 = 50
事务A：回滚
```

此时事务B读取到的数据 **50** 实际上是无效数据，因为事务A最终回滚了。

2. 不可重复读

不可重复读是指 **同一个事务多次读取同一数据，结果却不同**。

例如：

```
事务A：读取库存 = 100

事务B：修改库存 → 90 并提交

事务A：再次读取库存 = 90
```

在同一个事务中，两次读取的数据不一致。

3. 幻读

幻读是指 **同一个事务两次查询得到的记录数量不同**。

例如：

```
事务A：查询订单数量 = 10

事务B：插入一条订单并提交

事务A：再次查询订单数量 = 11
```

事务A第二次查询时，出现了新的记录，就像“幻影”一样。



为了解决并发访问带来的数据一致性问题，数据库引入了 **锁机制**。

**数据库锁是一种并发控制机制**，用于在多个事务同时访问同一数据时，通过对数据加锁的方式限制访问，从而保证数据的一致性和正确性。

> **锁的本质就是：在事务访问数据时，对数据资源进行占用控制，使得多个事务能够按照一定的顺序访问数据。**

通过锁机制，数据库可以：

- 防止多个事务同时修改同一条数据
- 保证事务执行的正确顺序
- 避免数据不一致问题

因此，**数据库锁是实现事务隔离性的核心机制之一**，也是数据库并发控制的重要基础。



## 二、数据库锁的分类

数据库为了在**数据一致性和系统并发性能之间取得平衡**，设计了多种不同类型的锁。从不同的角度来看，数据库锁可以有不同的分类方式。

通常可以从以下三个维度进行划分：

1. **按锁的粒度划分**：锁定数据范围的大小
2. **按锁的性质划分**：事务对数据的访问方式
3. **按 InnoDB 锁实现方式划分**：MySQL InnoDB 存储引擎的具体锁类型

### 2.1 按锁的粒度分类

锁的**粒度**指的是锁定数据的范围大小。锁的粒度越大，并发能力越低，但加锁成本越小；锁的粒度越小，并发能力越高，但加锁成本越大。

常见的锁粒度包括：**表锁、页锁、行锁**。

| 锁类型 | 锁定范围 | 并发性能 | 适用场景 |
| ------ | -------- | -------- | -------- |
| 表锁   | 整张表   | 低       | 批量操作 |
| 页锁   | 一页数据 | 中       | 较少使用 |
| 行锁   | 单行数据 | 高       | OLTP系统 |

#### 1 表锁

表锁是指对**整张数据表加锁**。当一个事务对表加锁后，其他事务在锁释放之前无法对该表进行相应操作。

特点：

- 加锁开销小
- 实现简单
- 并发能力较低

例如：

```
LOCK TABLE user WRITE;
```

> 在 MySQL 中，**MyISAM 引擎主要使用表锁**。

#### 2 页锁

页锁是指对**数据库中的一个数据页进行加锁**，一页通常包含多条记录。

页锁的粒度介于表锁和行锁之间：

- 并发能力高于表锁
- 加锁成本低于行锁

不过在 MySQL 中，页锁使用**较少**。

#### 3 行锁

行锁是指**只锁定某一条记录**，是粒度最小的锁。

特点：

- 并发能力高
- 锁冲突概率低
- 实现复杂

例如：

```
SELECT * FROM user WHERE id = 1 FOR UPDATE;
```

在 MySQL 中，**InnoDB 引擎默认使用行锁**。



### 2.2 按锁的性质分类

按照事务对数据的访问方式，数据库锁可以分为：

- **共享锁（Shared Lock）**
- **排他锁（Exclusive Lock）**


#### 1 共享锁

共享锁也称为**读锁（S锁）**。
 当一个事务对数据加上共享锁后，其他事务仍然可以读取该数据，但不能修改。

共享锁的特点：

| 操作    | 是否允许 |
| ------- | -------- |
| 读 + 读 | 允许     |
| 读 + 写 | 不允许   |

例如：

```
SELECT * FROM user LOCK IN SHARE MODE;
```


#### 2 排他锁

排他锁也称为**写锁（X锁）**。
 当一个事务对数据加上排他锁后，其他事务既不能读取，也不能修改该数据。

排他锁的特点：

| 操作    | 是否允许 |
| ------- | -------- |
| 写 + 写 | 不允许   |
| 写 + 读 | 不允许   |

例如：

```
SELECT * FROM user WHERE id = 1 FOR UPDATE;
```

### 2.3 InnoDB锁类型

在 MySQL 的 **InnoDB 存储引擎**中，为了支持高并发事务处理，引入了更加精细化的锁机制。常见的锁类型包括：

- **Record Lock（记录锁）**
- **Gap Lock（间隙锁）**
- **Next-Key Lock（临键锁）**

这些锁主要用于解决 **幻读问题**。

#### 1 Record Lock

记录锁是指**锁定某一条具体的记录**。

例如：

```
SELECT * FROM order WHERE id = 10 FOR UPDATE;
```

此时数据库只会锁住：

```
id = 10
```

这一条记录。


#### 2 Gap Lock

间隙锁是指锁住**记录之间的区间**，而不是具体的记录。

例如数据：

```
10   20   30
```

如果加间隙锁：

```
(10 , 20)
```

那么其他事务就**无法在这个区间插入数据**。

作用：

> 防止其他事务在查询范围内插入新数据。

#### 3 Next-Key Lock

Next-Key Lock 是 **Record Lock + Gap Lock 的组合**。

例如锁住：

```
(10 , 20]
```

既锁住：

- 记录 20
- 区间 (10,20)

这种锁是 **InnoDB 在 Repeatable Read 隔离级别下的默认锁机制**。
> 之前面试被问过临键锁的区间开闭：**左开右闭**

主要作用是：

> 防止幻读问题。

数据库锁通过不同的粒度和类型，在系统并发性能与数据一致性之间做出了权衡。粒度越小，并发能力越高；锁的控制越严格，数据一致性越强。在 MySQL 的 InnoDB 引擎中，通过 Record Lock、Gap Lock 和 Next-Key Lock 等机制，实现了高并发环境下的事务隔离与数据安全。



## 三、数据库加锁流程

在 MySQL 的 **InnoDB 存储引擎**中，数据库并不会在事务开始时立即对所有数据加锁，而是在执行 SQL 语句访问数据时 **按需加锁**。
 也就是说，只有当事务真正访问某条数据时，数据库才会对该数据加锁。数据库加锁的整体流程大致如下：

### 1. SQL语句执行

假设执行如下 SQL：

```
SELECT * FROM order WHERE id = 10 FOR UPDATE;
```

该语句表示查询 id = 10 的订单，同时对该记录加 **排他锁（X锁）**

### 2. 通过索引定位数据

InnoDB 在执行 SQL 时，首先会根据查询条件查找索引，数据库会通过 **B+Tree 索引**定位到对应的数据记录。如果查询条件没有索引，数据库可能需要 **全表扫描**。

### 3. 判断是否需要加锁

当 SQL 语句包含以下操作时，数据库通常会进行加锁：

| SQL类型           | 是否加锁       |
| ----------------- | -------------- |
| SELECT            | 默认不加锁     |
| SELECT FOR UPDATE | 加排他锁       |
| UPDATE            | 加排他锁       |
| DELETE            | 加排他锁       |
| INSERT            | 对插入位置加锁 |

### 4. 对数据记录加锁

当数据库定位到目标记录后，就会在 **锁管理器（Lock Manager）** 中记录该锁信息。

例如：

```
事务A
锁类型：X锁
锁对象：id = 10
```

此时如果其他事务也尝试访问该记录：

```
事务B
UPDATE order SET status=1 WHERE id=10
```

事务B就会：

```
等待锁释放
```

### 5. 执行数据操作

加锁成功后，事务就可以安全地对数据进行操作，例如：

```
读取数据
更新数据
删除数据
```

由于锁的存在，其他事务无法对该记录进行冲突操作，从而保证了数据的一致性。

### 6. 释放锁

事务结束后，数据库会自动释放该事务持有的所有锁，释放锁之后其他等待的事务可以继续执行。
